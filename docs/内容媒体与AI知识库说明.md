# 内容、媒体与 AI 知识库说明

更新日期：2026-06-21

## 一、现在已经实现的管理员能力

管理员可以在后台实时维护三类记录：

| 类型 | 新增 | 编辑 | 删除 | 文本 | 图片 | 文档导入 |
| --- | --- | --- | --- | --- | --- | --- |
| 学习 | 支持 | 支持 | 支持 | 支持 Markdown | 支持 JPEG、PNG、WebP | 支持 `.md`、`.markdown`、`.txt` |
| 生活 | 支持 | 支持 | 支持 | 支持 Markdown | 支持 JPEG、PNG、WebP | 不需要 |
| 工作 | 支持 | 支持 | 支持 | 支持 Markdown | 支持 JPEG、PNG、WebP | 不需要 |

入口：

- `/admin/study`
- `/admin/life`
- `/admin/work`

每条记录都可以设置：

- 状态：草稿、发布、归档。
- 可见范围：私密、不公开列出、公开。
- 是否允许 AI 引用文本。
- 标题、摘要、正文、标签及类型专属字段。

删除是永久删除。系统同时删除该内容生成的知识条目和知识分块；与该内容绑定且没有被其他内容使用的图片、缩略图和媒体数据库记录也会清理。编辑记录时也可以单独替换或移除当前图片。

## 二、文本和图片分别存在哪里

### 1. 文本与结构化数据

文本保存在 PostgreSQL：

| 表 | 保存内容 |
| --- | --- |
| `content_items` | 工作、生活、学习记录的标题、摘要、正文、状态、可见性和业务字段 |
| `knowledge_items` | AI 可维护的纯文本知识副本、手工知识和访客问题知识 |
| `knowledge_chunks` | 对知识正文切分后的文本块、Token 估算和可选向量 |
| `chat_sessions` | 访客会话 |
| `chat_messages` | 访客消息、AI 回复和引用来源 |
| `visitor_questions` | 访客问题、原回答、主题、状态和转换结果 |
| `ai_call_logs` | 模型、耗时、Token、成功状态和错误码 |
| `admin_audit_logs` | 管理员新增、更新、删除、上传和知识转换记录 |

### 2. 图片二进制文件

图片文件不存进 PostgreSQL。数据库只保存媒体索引：

| 表 | 保存内容 |
| --- | --- |
| `media_assets` | 存储键、文件名、MIME、尺寸、大小、校验值、替代文本、原图/缩略图关系 |
| `content_media` | 哪条内容绑定了哪张原图，以及显示顺序 |

实际图片存储位置由 `STORAGE_DRIVER` 决定：

- `local`：保存到项目的 `storage/uploads/`，适合本地开发。
- `s3`：保存到 S3 兼容对象存储，适合正式部署，可对接 AWS S3、腾讯云 COS、阿里云 OSS 等。

上传一张图片时，系统会：

1. 检查文件真实类型，只允许 JPEG、PNG、WebP。
2. 限制单文件最大 15 MB、总像素不超过 8000 万。
3. 保存原图。
4. 自动生成最长边不超过 800 像素的 WebP 缩略图。
5. 计算 SHA-256 校验值。
6. 在数据库写入原图和缩略图索引。
7. 把原图绑定到内容记录。

## 三、私人图片如何保护

图片和 AI 知识库完全分离：

- AI 检索只查询 `knowledge_items.body` 和文本标签。
- AI 组装上下文时不查询 `media_assets`、`content_media` 或对象存储。
- 图片文件名、存储路径、EXIF、替代文本和二进制内容不会发送给 DeepSeek。
- 内容即使上传了图片，也只有显式授权的文本会生成知识副本。

图片访问还受内容权限控制：

- 管理员登录后可以读取属于自己的媒体。
- 访客只有在图片绑定到“已发布 + 公开”的内容时才能读取。
- 草稿、私密、不公开列出或未绑定的图片，访客访问媒体 API 会得到拒绝或不存在。

如果一张照片只想自己保留：

1. 将内容设为“私密”。
2. 不勾选“允许 AI 引用文本”。
3. 图片仍可在后台维护，但不会公开，也不会进入模型上下文。

## 四、内容如何自动进入 AI 知识库

保存工作、生活或学习记录时，后台在同一个数据库事务中执行：

```text
保存 content_items
  → 提取允许使用的纯文本字段
  → 新建或更新 knowledge_items
  → 删除旧 knowledge_chunks
  → 按约 800 字、100 字重叠重新分块
  → 写入新的 knowledge_chunks
  → 写入管理员审计日志
```

不同内容进入知识库的文本字段：

| 类型 | 会进入知识库的字段 |
| --- | --- |
| 学习 | 标题、摘要、正文、类型、学习周期、机构或来源、标签 |
| 生活 | 标题、摘要、正文、地点、心情、标签 |
| 工作 | 标题、摘要、正文、个人角色、成果、项目周期、技术栈、标签 |

不会进入知识库：

- 图片原图与缩略图。
- 图片文件名和存储键。
- 媒体尺寸、校验值和其他媒体元数据。
- 后台私密字段。

只有以下三个条件同时成立，知识才允许用于访客回答：

```text
status = published
visibility = public
isAiUsable = true
```

将内容改为草稿、归档、私密，或关闭 AI 引用后，对应知识会同步禁用。

现有历史内容可执行：

```powershell
pnpm db:sync-knowledge
```

脚本会把当前工作、生活、学习记录重新同步成纯文本知识，并重建分块。

## 五、模型如何拿到相关数据

访客提问后的实际流程：

1. 创建或复用聊天会话。
2. 保存访客消息。
3. 根据问题和场景在 PostgreSQL 中搜索知识。
4. 访客检索只允许以下来源：
   - `study_item`
   - `work_project`
   - `life_record`
   - `visitor_question`
5. 再强制检查知识必须为“已发布 + 公开 + AI 可用”。
6. 当前主检索使用 PostgreSQL `pg_trgm`，比较标题、分类、正文和标签的文本相似度。
7. 选出最相关条目，生成纯文本上下文和引用列表。
8. 把以下内容发送给 DeepSeek：
   - 系统提示词和隐私规则。
   - 当前咨询场景。
   - 检索出的纯文本知识。
   - 最近 6 条对话。
   - 当前访客问题。
9. 保存 AI 回复、引用来源、模型信息和耗时。
10. 另存一条 `visitor_questions`，供管理员后续筛选。

如果 DeepSeek 未配置或调用失败，系统使用同一批纯文本知识生成本地降级回答，不会因此读取图片。

pgvector 已安装，`knowledge_chunks.embedding` 也是正式向量字段并带 HNSW 索引；当前在线检索仍以 `pg_trgm` 为主。后续接入 embedding 任务后，可以升级为“向量召回 + 文本召回”的混合检索，不需要改变图片隐私边界。

## 六、如何保留访客感兴趣的问题

每次真实提问都会留下：

- 问题原文。
- AI 原回答。
- 所属主题。
- 本次引用的知识来源。
- 会话和消息关联。
- 创建时间。
- 当前处理状态。

管理员操作：

1. 进入 `/admin/questions` 查看访客问题。
2. 对值得长期保留的问题点击“问题转知识”。
3. 系统创建来源为 `visitor_question` 的知识条目，并保留原问题关联。
4. 进入 `/admin/knowledge` 编辑标题、分类、标准回答、标签、状态、可见性和 AI 引用开关。
5. 更新知识正文后，系统自动重新分块；停用后 AI 立即不再引用。

这样既保留了“大家实际关心什么”的记录，也允许管理员在正式进入知识库前校对答案。

## 七、正式部署时的推荐存储方案

### 推荐架构

```text
Next.js 网站
  ├─ 托管平台：Vercel 或 Node.js 服务器
  ├─ PostgreSQL：托管 PostgreSQL，启用 pg_trgm 和 vector
  ├─ 图片：私有 S3/COS/OSS Bucket
  └─ DeepSeek：服务端环境变量保存 API Key
```

### 为什么部署时不能继续使用 local

Vercel 等无服务器平台的本地文件系统不是持久存储。实例重启、扩缩容或重新部署后，写入 `storage/uploads/` 的图片可能消失；多个实例之间也不会共享文件。

因此正式部署必须设置：

```env
STORAGE_DRIVER=s3
S3_REGION=对象存储区域
S3_BUCKET=Bucket名称
S3_ENDPOINT=https://对象存储API地址
S3_ACCESS_KEY_ID=访问密钥ID
S3_SECRET_ACCESS_KEY=访问密钥Secret
S3_FORCE_PATH_STYLE=false
```

建议 Bucket 保持私有，不要开放公共读。网站通过 `/api/media/{id}` 在服务端检查管理员会话或内容公开状态后，再从对象存储读取并返回文件。

### PostgreSQL 部署要求

- 使用持久化托管 PostgreSQL，不要依赖应用容器内部数据库。
- 数据库需要安装 `pg_trgm` 和 `vector` 扩展。
- 设置 `DATABASE_URL`、连接数和 SSL。
- 部署前执行 `pnpm db:migrate`。
- 首次部署或重置管理员时执行 `pnpm db:seed`。
- 导入历史内容后执行 `pnpm db:sync-knowledge`。

### 备份范围

完整备份必须同时包含：

1. PostgreSQL 数据库备份。
2. 对象存储 Bucket 的版本控制或生命周期备份。

仅备份数据库无法恢复图片；仅备份 Bucket 无法恢复内容关系、知识库和聊天记录。

## 八、已验证结果

本地端到端验证已覆盖：

- 创建公开生活记录。
- 上传 PNG 原图并生成缩略图。
- 绑定图片并在访客 API 返回图片地址。
- 自动生成纯文本知识。
- 确认知识正文不含图片文件名或存储路径。
- 更新内容后知识同步更新。
- 删除内容后知识、媒体数据库记录和实际文件同步清理。
- 私密学习记录的图片：管理员读取返回 `200`，未登录访客读取返回 `404`。
- 私密记录的知识副本保持 `private + AI 不可用`。
- 真实访客提问会保存问题和 5 条引用；管理员可转换为知识并修改为校对后的标准答案。
