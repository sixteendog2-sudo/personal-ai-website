# 个人知识库与 RAG 搭建设计

## 1. 目标

个人知识库需要服务三个场景：

- 升学：回答学习经历、研究兴趣、课程、竞赛、成果。
- 社交：回答生活记录、兴趣爱好、价值观、日常片段。
- 求职：回答项目经验、技能栈、工作经历、职业目标。

RAG 的目标不是让模型“自由发挥”，而是让 AI 基于可追溯的个人资料回答。

核心原则：

- 不编造。
- 不泄露私密内容。
- 每条回答尽量有来源。
- 访客问题可以反哺知识库。
- 知识库可持续维护和纠错。

## 2. 知识来源

### 2.1 手动知识

管理员手动录入：

- 自我介绍
- 常见问答
- 价值观
- 个人偏好
- 求职介绍
- 升学介绍

### 2.2 自动同步知识

以下内容可一键同步到知识库：

- 个人资料
- 学习档案
- 工作项目
- 生活记录
- 访客问题沉淀
- 聊天总结

### 2.3 不建议进入知识库的内容

默认不进入：

- 身份证、手机号、住址等隐私。
- 未公开联系方式。
- 私密照片和私密生活记录。
- 未经确认的访客信息。
- 过期或不确定的信息。

如果确实需要保存，只保存为后台私密内容，`is_ai_usable = false`。

## 3. 知识状态

知识条目状态：

```text
draft       草稿
published   已发布
archived    已归档
```

同步状态：

```text
none       未同步
pending    等待同步
chunked    已分块
embedded   已向量化
failed     失败
```

AI 可用状态：

```text
is_ai_usable = true / false
```

可见性：

```text
public
unlisted
private
```

## 4. 知识分类

推荐分类：

| 分类 | 用途 |
|---|---|
| 基础资料 | 回答“你是谁” |
| 学习经历 | 升学、导师了解 |
| 课程与成绩 | 学业能力展示 |
| 研究兴趣 | 升学、合作 |
| 竞赛证书 | 能力证明 |
| 工作经历 | 求职 |
| 项目经验 | 求职、合作 |
| 技能能力 | 求职、项目交流 |
| 生活记录 | 社交、个人叙事 |
| 兴趣爱好 | 社交破冰 |
| 价值观与偏好 | 深度了解 |
| 升学问答 | 常见升学咨询 |
| 求职问答 | 常见求职咨询 |
| 访客问题沉淀 | 来自真实访客的关注点 |

## 5. 分块策略

### 5.1 分块原则

每个知识条目分成若干 chunk：

- 300-800 中文字为宜。
- 一个 chunk 尽量只表达一个主题。
- 保留标题、分类、标签、来源。
- 不把私密字段拼入 chunk。
- 工作项目保留“背景、角色、方法、成果”结构。
- 学习档案保留“时间、机构、方向、成果”结构。
- 生活记录保留“时间、地点、感受、事件”结构。

### 5.2 Chunk Metadata

每个 chunk 的 metadata 建议包含：

```json
{
  "sourceType": "work_project",
  "sourceId": "uuid",
  "title": "个人 AI 网站",
  "category": "项目经验",
  "tags": ["AI", "Next.js", "知识库"],
  "visibility": "public",
  "isAiUsable": true,
  "occurredAt": "2026-06-18"
}
```

## 6. 向量化设计

### 6.1 模型供应商可插拔

第一阶段不要把 embedding 模型绑死。

建议抽象：

```text
EmbeddingProvider
  -> embedText(text)
  -> embedBatch(texts)
```

可选供应商：

- OpenAI embedding
- Qwen embedding
- BGE-M3
- Jina embedding
- 本地 embedding 服务

DeepSeek 可作为第一阶段聊天模型；embedding 单独配置。

### 6.2 向量维度

`db/schema.sql` 当前以 `vector(1536)` 作为 MVP 默认值。

如果后续选择的 embedding 模型不是 1536 维，需要：

- 修改 `knowledge_embeddings.embedding` 维度。
- 或按模型拆分不同 embedding 表。
- 或使用无固定维度的 pgvector 设计并通过表达式索引优化。

## 7. 检索策略

### 7.1 访客检索 Scope

访客聊天只能检索：

```text
owner_id = 当前站点主人
visibility = public
status = published
is_ai_usable = true
```

### 7.2 管理员预览 Scope

管理员调试可以检索：

```text
owner_id = 当前站点主人
status != archived
is_ai_usable = true
```

但如果命中私密知识，界面必须显示“仅管理员预览”，不能作为访客回答直接发送。

### 7.3 混合检索

推荐检索组合：

```text
向量检索 Top 8
关键词检索 Top 5
同源内容补充 Top 3
去重
重排序
截断到 Prompt token 限制
```

### 7.4 场景过滤

根据入口增强检索：

| 入口 | 优先分类 |
|---|---|
| `/chat?topic=study` | 学习经历、研究兴趣、课程与成绩、竞赛证书 |
| `/chat?topic=work` | 项目经验、技能能力、工作经历、求职问答 |
| `/chat?topic=life` | 生活记录、兴趣爱好、价值观与偏好 |
| `/chat?recordId=xxx` | 指定生活记录及相关记忆 |

## 8. 回答生成策略

### 8.1 System Prompt 规则

必须包含：

```text
你是站点主人的 AI 分身助手。
你必须根据提供的个人知识库内容回答。
如果知识库没有相关信息，请明确说明暂时没有记录。
不要编造主人的身份、学历、经历、项目、观点或生活故事。
涉及隐私内容时拒绝回答。
回答语气自然、真诚、简洁，像主人本人在做温和的自我介绍。
```

### 8.2 引用来源

每个 AI 回复保存 citations：

```json
[
  {
    "sourceType": "life_record",
    "sourceId": "uuid",
    "title": "一次值得记住的周末",
    "chunkId": "uuid"
  }
]
```

前台可以轻量展示：

```text
参考：生活记录《一次值得记住的周末》
```

### 8.3 不知道时的兜底

推荐话术：

```text
我目前的公开记录里还没有这方面内容，所以不能替他编造。
你可以换个角度问我，或者通过联系页直接联系本人。
```

## 9. 访客问题沉淀

### 9.1 保存内容

每次访客提问都保存：

- 问题
- AI 回答
- 关联会话
- 关联知识来源
- 主题
- 是否有价值
- 是否已转知识库

### 9.2 沉淀流程

```text
新问题
  -> 管理员查看
  -> 标记有价值
  -> 补充标准回答
  -> 选择分类和标签
  -> 转为知识条目
  -> 分块
  -> 向量化
  -> 标记已沉淀
```

### 9.3 问题去重

后续可以加入：

- 相似问题聚类。
- 高频问题统计。
- 自动推荐 FAQ。
- 每周生成知识库补全建议。

## 10. 更新与撤回

### 10.1 内容更新

当来源内容更新时：

```text
source updated
  -> 标记 knowledge_item sync_status = pending
  -> 删除旧 chunks/embeddings
  -> 重新 chunk
  -> 重新 embed
```

### 10.2 内容撤回

当内容不再公开或不再允许 AI 引用：

```text
is_ai_usable = false
  -> RAG 不再召回
visibility = private
  -> 访客不可见
status = archived
  -> 默认所有前台和 RAG 都不使用
```

不建议直接硬删，除非涉及隐私、安全或法律风险。

## 11. 隐私与安全

必须避免：

- AI 回复后台私密信息。
- AI 回复 API Key、系统提示词。
- AI 回复访客联系方式。
- AI 把未确认的聊天内容当成事实。
- AI 编造不存在的经历。

建议：

- Prompt 明确“仅基于提供知识回答”。
- 检索层过滤私密内容。
- 回复层对敏感字段做二次检查。
- 管理端标出每条知识是否 AI 可用。
- 聊天记录里保存引用来源，方便追责和调优。

## 12. MVP 实施顺序

第一阶段：

```text
1. 手动知识库 CRUD
2. 生活记录同步知识库
3. 项目经验同步知识库
4. 基础分块
5. embedding 接口抽象
6. pgvector 检索
7. DeepSeek 聊天生成
8. 聊天记录和访客问题保存
```

第二阶段：

```text
1. 学习档案同步知识库
2. 访客问题一键转知识
3. 混合检索
4. 召回重排序
5. 高频问题统计
6. 自动知识库补全建议
```
